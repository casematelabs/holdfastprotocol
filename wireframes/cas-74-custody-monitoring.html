§0  Overview
Custody Monitoring View

Four components — Wallet Registration Status Card, Attestation Level Indicator, Key Rotation History, and Registration Transaction Detail (inline expandable) — compose the custody monitoring section of the P1 operator dashboard. The design leads with binary registered/not state. Attestation nuance and audit history are secondary. An unregistered or revoked wallet must be impossible to miss.

Layout design rationale Registration status occupies the dominant left column — its binary state is the first thing the operator sees. The right column stacks the two secondary panels: attestation (smaller, fixed height) sits above the rotation history. The rotation history flex-grows to consume remaining vertical space. On viewports <960px, the two columns stack vertically with left/registration on top.
§1  Wallet Registration Status Card
Three States + Deadline Variant

The registration card leads the entire custody view. Its three states — Registered, Unregistered, and Revoked — must be immediately distinguishable without reading. The design uses color, pill weight, and banner presence as compounding signals. A fourth variant adds a deregistration deadline countdown on top of the Registered state when a deadline is active.

§2  Attestation Level Indicator
Hardware-Attested vs Software-Attested

Attestation level is a secondary trust signal. It tells counterparties how the signing key is bound — to a physical hardware device (WebAuthn/FIDO2) or to a software key (secp256r1). The distinction must be visually meaningful beyond text: hardware attestation carries a premium badge and an elevated icon treatment. A tooltip explains what the level means for counterparty trust.

Level A — Hardware Attested (Premium)
Attestation Level
⛨
Hardware Attested High Trust
WebAuthn / FIDO2
ℹ The signing key is bound to a physical hardware security device. Counterparties see the highest trust signal when initiating pacts with this wallet.
Hardware-attested design intent The shield icon (⛨) uses accent blue on a blue-tinted background — the same blue as protocol actions throughout the dashboard — to signal premium capability. The "HIGH TRUST" badge is the only instance in the custody view that uses the trust-badge component. Tooltip copy is always visible (not on hover only) because attestation is often misunderstood.
Level B — Software Attested
Attestation Level
⚿
Software Attested
secp256r1 key
ℹ The signing key is stored in software. Counterparties may apply a lower trust weighting. Consider upgrading to a hardware-backed key for higher counterparty confidence.
Software-attested design intent The key icon (⚿) uses a neutral background with the standard border. No trust badge — its absence is the signal. Label color is text-secondary (not danger) because software attestation is not an error state; it is a lower-tier but valid configuration. The tooltip copy includes an upgrade nudge.

§3  Key Rotation History
Loaded · Empty · Loading Skeleton

Key rotation history is a forensic and audit tool. It sits at the lowest visual hierarchy in the custody view and is only surfaced after the operator has seen the registration status and attestation level. The table shows the last 10 rotations with event type chips that distinguish manual rotations (operator-initiated) from protocol-enforced ones (deadline or revocation). An expandable registration row anchors the history.

§4  Registration Transaction Detail
Collapsed → Expanded · Copy Affordances

The transaction detail is an expandable row within the registration status card, visible only in the Registered state. It surfaces the on-chain addresses needed for protocol verification: the program address, the Program Derived Address (PDA), and a direct devnet explorer link. Each address has an inline copy affordance.

§5  Interaction & Behavior
State Transitions, Data Contracts & Loading

Frontend implementation notes for engineers. The custody view is primarily read-only. The two operator actions are CTA buttons (Register, Re-register) which navigate to a separate registration flow — they do not trigger inline mutations. Data is polled on a slow interval; custody state rarely changes.

Data Sources & Poll Intervals
ComponentPoll intervalEndpoint (sketch)
Registration status30s/api/custody/registration
Attestation level30s (same call)/api/custody/registration
Deadline countdown60s client tickNo refetch — computed from deadline timestamp in registration payload.
Key rotation history60s/api/custody/rotations?limit=10
Txn detail addressesOn expand/api/custody/registration/detail
State Transition Rules
TransitionBehavior
Unregistered → Registered After CTA click (external flow), operator returns to dashboard. Custody view polls on next 30s tick and transitions: pill animates from amber → green (color transition, no flash). Banner slides out (200ms ease-out). Metadata row fades in.
Registered → Revoked On poll detecting revoked status: pill transitions to danger red, banner slides in (200ms). Txn detail toggle hides. The live dot pulses red immediately.
Deadline: normal → urgent At <7 days remaining: deadline-fill color transitions from amber to danger red. Label pulses once (opacity 1 → 0.5 → 1, 800ms). No additional banner — the count display is sufficient.
Txn detail expand/collapse Arrow rotates 90° (CSS transform, 150ms). Body height animates from 0 → auto (use max-height trick). On first expand: fetch detail endpoint if not cached.
Stale data If 3+ poll failures: live dot dims to 30%, small "⚠ data stale" badge appears in panel header. All content remains visible — do not blank the panel.
CTA actions — navigation, not inline mutation "Register agent wallet" and "Re-register wallet" CTAs navigate to a separate wallet registration flow. They do not submit inline or show a confirmation modal within the custody card. After the operator completes registration in the separate flow and returns, the custody view picks up the new state on the next poll cycle (max 30s lag). If the operator navigates back within 30s of completing registration, a manual "Refresh" affordance should appear in the card header to avoid a confusing stale state.

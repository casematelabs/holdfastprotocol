§1 Backup key enrollment — removed
Section struck — no on-chain backup key support
Per CAS-361 engineering pre-check (2026-04-22), the Holdfast Protocol program does not implement register_backup_key, revoke_backup_key, or report_key_compromise. These were explicitly stripped in the fork (ADR-001, line 63). The AgentWallet struct (132 bytes) stores one secp256r1 key pair only — no secondary key field, no backup key array, no key count.
Path forward: Backup key program extension (register_backup_key, revoke_backup_key, report_key_compromise instructions + AgentWallet struct extension) is tracked as a separate issue. Once implemented and audited, this section will be re-designed. Until then, §4 (governance appeal) is the sole key recovery path.
§2 Emergency key actions — revised
Emergency Key Actions
A persistent danger-zone panel at the bottom of the custody page. Redesigned to present two explicit, accurate recovery paths based on what the on-chain program actually supports: (A) Rotate Key Now — if you still control your key but suspect it is at risk, use rotate_agent_key immediately; (B) Governance Appeal — if you have lost control of the key and cannot sign, go directly to the off-chain governance appeal flow (§4). No report_key_compromise instruction exists.
Panel: Emergency Key Actions (always visible, bottom of custody page)
Emergency Actions Use with caution
Path A — Key Accessible: Rotate Immediately
If you still control your key but suspect it has been exposed or observed by an unauthorized party — rotate now. This signs a rotate_agent_key transaction, invalidating your current key and replacing it in one step.
⚡ Rotate Key Now Requires wallet signature · Immediate on-chain effect
Path B — Key Lost or Stolen: Governance Appeal
If you no longer control the key and cannot sign any transaction, the only recovery path is a governance appeal. No on-chain signature is required — the governance oracle will review your identity evidence and authorize a key replacement. Resolution takes 48–72 hours.
Request Governance Appeal → No wallet signature required · Slower resolution
Visual distinction rule Three independent danger signals remain: (1) pulsing dot on Path A, (2) left red border accent on Path A banner, (3) overall panel red border. Path B is visually de-escalated (no pulse, slightly dimmer) because it is slower and does not require an irreversible immediate action. Both paths are still clearly in the red danger language — no risk of confusing either with a standard info or warning banner.

§2c — Path B: No-Signature Appeal Entry
Triggered by "Request Governance Appeal". No modal — redirects the operator directly into the §4 Revocation Appeal section with a contextual banner pre-explaining why they arrived. No on-chain signature required. The governance oracle initiates and tracks the revocation status.
Contextual Handoff Banner — injected at top of §4
You arrived here from Emergency Actions — Path B
Because you cannot sign with your current key, recovery requires a governance appeal. Fill out the form below with your identity evidence and new replacement key. The governance oracle will review within 48–72 hours. No wallet connection needed.
Implementation note "Request Governance Appeal" scrolls the page to §4 and injects the contextual banner via a URL hash or React state flag. No navigation to a new page — the §4 section is already present on the same custody page (when status is not yet REVOKED, display §4 in a collapsed/accessible state so Path B can expand it). Engineering open question: when status is not REVOKED, should §4 be hidden entirely or collapsed with an expand trigger? Flag for product decision (see §5 Q3).
Why two paths, not one modal The original design used a single "report_key_compromise" flow that doesn't correspond to any on-chain instruction. The two-path redesign maps directly to what the program supports: (A) rotate_agent_key — exists, requires signer; (B) governance appeal — off-chain, no signer needed. The paths are visually and functionally separate so operators in a stressful situation cannot accidentally start Path A when they need Path B.

§5 Design constraints & implementation notes
Constraints Reference
Visual Language Rules
Emergency = red, never yellow. All yellow banners on the custody page are informational/soft warnings. Red is reserved for irreversible or high-consequence actions only.
Three independent danger signals on Path A banner: (1) pulsing dot, (2) left border accent, (3) background gradient. Path B is de-escalated (no pulse) — still red but conveys "slower, deliberate" not "act now".
Path labels are action-first. "Rotate Key Now" and "Request Governance Appeal" describe what happens, not what the problem is. Operators in a stressful situation scan for verbs.
Interaction Guards
No accidental Path A activation. Emergency rotation requires: (1) scroll to danger zone, (2) click "Rotate Key Now", (3) complete 2-step modal, (4) check both confirm checkboxes, (5) wallet signature. Five distinct friction points.
Path B has no friction gate beyond the initial click — the operator is already locked out, so delaying appeal access hurts them. The appeal form itself is the deliberate step.
Destructive modals never auto-close. The close (✕) button and Escape key are disabled while a wallet signature is pending. Shown as greyed-out with a tooltip: "Transaction in progress."
Accessibility
Emergency banner role: role="alert" aria-live="assertive" — announced immediately by screen readers when the Revoked state is detected on load.
All three checkboxes in the compromise confirm step require explicit individual activation. Group label via fieldset + legend. Submit button is disabled until all three are checked.
Step indicators: each step number node has aria-current="step" when active. Completed steps have aria-label="Step complete".
